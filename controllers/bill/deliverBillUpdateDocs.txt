FUNCTION NAME:
deliverBillUpdate

DESCRIPTION:
This API is one of the most important functions in the system as it handles
the complete delivery process for multiple bills. When bills are marked as
"Delivered", it triggers several critical business operations:

1. Updates bill status to "Delivered"
2. Adjusts inventory (moves from reserved to sold)
3. Creates stock transaction logs
4. Updates retailer ledger with debit/credit entries
5. Processes RBP (Reward Based Program) point transactions
6. Handles API calls to external retailer reward system

This function ensures complete business workflow completion from order to delivery.

------------------------------------------------------------
STEP 1: Input Validation and Authentication
------------------------------------------------------------
1.1 Extract Request Data:
    - billIds: Array of bill IDs to be marked as delivered
    - req.user: Authenticated distributor information

1.2 Validate billIds Array:
    - Must be an array and cannot be empty
    - If invalid → return HTTP 400 with error:
      "Invalid or empty billIds array"

1.3 Validate User Authentication:
    - Check if req.user and req.user._id exist
    - If missing → return HTTP 401 with error:
      "Unauthorized: user not found"

1.4 Remove Duplicate Bill IDs:
    - Use Set to eliminate duplicate entries:
      const uniqueBillIds = [...new Set(billIds)]
    - This prevents processing the same bill multiple times

------------------------------------------------------------
STEP 2: Update Bill Status to "Delivered"
------------------------------------------------------------
2.1 Bulk Update Operation:
    - Use Bill.updateMany() to update all bills at once:
      Bill.updateMany(
        { _id: { $in: uniqueBillIds } },
        { 
          $set: { 
            status: "Delivered", 
            "dates.deliveryDate": new Date() 
          }
        }
      )

2.2 Validation Check:
    - If billStatus.matchedCount === 0:
      → return HTTP 404 with error: "No bills found to Deliver"
    - This means none of the provided bill IDs exist in the database

------------------------------------------------------------
STEP 3: Fetch Updated Bills with Population
------------------------------------------------------------
3.1 Retrieve Complete Bill Data:
    - Fetch all delivered bills with populated references:
      Bill.find({ _id: { $in: uniqueBillIds } }).populate(
        "lineItems.product lineItems.inventoryId"
      )
    - Population ensures we have complete product and inventory details

3.2 Data Structure After Population:
    - Each bill contains complete lineItems with:
      * product: Full product document
      * inventoryId: Full inventory document
      * billQty: Quantity delivered

------------------------------------------------------------
STEP 4: INVENTORY UPDATES (Critical Business Logic)
------------------------------------------------------------
For each bill in the delivered bills list:

4.1 Skip Empty Bills:
    - If bill has no lineItems or empty array, skip to next bill

4.2 Process Each Line Item:
    For each item in bill.lineItems:

    4.2.1 Basic Validations:
          - Skip if inventoryId or product is missing
          - Fetch current inventory: Inventory.findById(item.inventoryId._id)
          - Skip if inventory not found

    4.2.2 Calculate Current Stock Totals:
          - prevAvailableQty = inventory.availableQty || 0
          - prevReservedQty = inventory.reservedQty || 0  
          - totalQty = prevAvailableQty + prevReservedQty
          - Skip if totalQty <= 0 (avoid division by zero)

    4.2.3 Validate Bill Quantity:
          - billQty = Number(item.billQty) || 0
          - Skip if billQty <= 0
          - Skip if inventory.reservedQty < billQty (prevent negative reserved stock)

    4.2.4 Calculate Average Stock Values:
          - avgDlp = Math.round(inventory.totalStockamtDlp / totalQty)
          - avgRlp = Math.round(inventory.totalStockamtRlp / totalQty)
          These represent average cost per unit for DLP and RLP pricing

    4.2.5 Update Inventory Quantities:
          CRITICAL OPERATION - Moving stock from "reserved" to "sold"
          - inventory.reservedQty -= billQty
          - inventory.totalStockamtDlp -= billQty * avgDlp  
          - inventory.totalStockamtRlp -= billQty * avgRlp
          - await inventory.save()

    4.2.6 Create Stock Transaction Log:
          - balanceCount = totalQty - billQty
          - Create new Transaction record:
            {
              distributorId: req.user._id,
              productId: item.product._id,
              invItemId: inventory._id,
              qty: billQty,
              transactionId: await transactionCode("LXSTA"),
              date: new Date(),
              type: "Out", // Stock going out
              balanceCount,
              description: `Delivered against Bill: ${bill.billNo}`,
              transactionType: "delivery",
              stockType: "salable"
            }

INVENTORY SUMMARY:
- Reserved stock becomes sold stock
- Stock value is proportionally reduced
- Complete audit trail is maintained via Transaction log

------------------------------------------------------------
STEP 5: LEDGER UPDATES (Financial Accounting)
------------------------------------------------------------
5.1 Extract Financial Data:
    - creditAmount = Number(bill.creditAmount) || 0
    - This represents any credit adjustments applied to the bill

5.2 Get Latest Ledger Balance:
    - Find the most recent ledger entry for this distributor-retailer pair:
      Ledger.findOne({
        dbId: req.user._id,
        retailerId: bill.retailerId
      }).sort({ createdAt: -1 })
    - latestLedgerBalance = latestLedger ? Number(latestLedger.balance) : 0

5.3 Create Main Sales Debit Entry:
    - Generate unique transaction ID: 
      transactionId = await ledgerTransactionCode("LEDG", req.user._id)
    - Calculate new balance: newBalance = latestLedgerBalance - bill.netAmount
    - Create ledger entry:
      {
        dbId: req.user._id,
        retailerId: bill.retailerId,
        transactionId,
        transactionType: "debit",
        transactionFor: "Sales",
        billId: bill._id,
        transactionAmount: Number(bill.netAmount),
        balance: Number(newBalance.toFixed(2))
      }

5.4 Handle Credit Adjustments (if applicable):
    - Add 200ms delay to avoid race conditions:
      await new Promise((resolve) => setTimeout(resolve, 200))
    
    - If creditAmount > 0:
      * Fetch latest ledger balance again (after the debit entry)
      * Generate new transaction ID for credit
      * Create credit adjustment entry:
        {
          dbId: req.user._id,
          retailerId: bill.retailerId,
          transactionId: creditTransactionId,
          transactionType: "credit",
          transactionFor: "Sales-Credit-Adjustment",
          billId: bill._id,
          transactionAmount: creditAmount,
          balance: Number((latestLedgerBalance2 + creditAmount).toFixed(2))
        }

LEDGER SUMMARY:
- Sales amount is debited (reduces retailer's credit balance)
- Any credit adjustments are applied separately
- Complete financial audit trail is maintained

------------------------------------------------------------
STEP 6: RBP SCHEME VALIDATION
------------------------------------------------------------
6.1 Check RBP Scheme Mapping:
    - Validate if req.user.RBPSchemeMapped === "yes"
    - If not mapped:
      * Log message: "RBP scheme not mapped for distributor: {dbCode}"
      * Skip RBP processing and continue to next bill
    - This ensures only eligible distributors can earn/transfer RBP points

------------------------------------------------------------
STEP 7: RBP POINTS CALCULATION AND PROCESSING
------------------------------------------------------------
7.1 Get Current RBP Balance:
    - Find latest distributor transaction:
      DistributorTransaction.findOne({
        distributorId: req.user._id
      }).sort({ createdAt: -1 })
    - latestBalance = latestTransaction ? latestTransaction.balance : 0

7.2 Get Retailer Information:
    - retailer = await OutletApproved.findById(bill.retailerId)
    - retailerUID = retailer?.outletUID || ""
    - dbCode = req.user.dbCode

7.3 Calculate Total Reward Points:
    - Initialize: totalRewardPoints = 0
    - For each item in bill.lineItems:
      * Get product: Product.findById(item.product)
      * Extract base_point: Number(productData.base_point) || 0
      * If base_point > 0:
        totalRewardPoints += base_point * Number(item.billQty)

    Example calculation:
    Product A: base_point = 2, billQty = 5 → 10 points
    Product B: base_point = 3, billQty = 2 → 6 points  
    Total = 16 points

------------------------------------------------------------
STEP 8: RBP BALANCE VALIDATION AND API PROCESSING
------------------------------------------------------------
8.1 Check Sufficient RBP Balance:
    - If totalRewardPoints > latestBalance OR latestBalance <= 0:
      * Log insufficient balance message
      * Create failed transaction record:
        {
          distributorId: req.user._id,
          transactionType: "debit",
          transactionFor: "SALES", 
          point: totalRewardPoints,
          balance: latestBalance, // No change to balance
          billId: bill._id,
          retailerId: bill.retailerId,
          status: "Failed",
          remark: "Insufficient RBP balance for Bill no {billNo}...",
          apiResponse: {
            error: true,
            message: "Insufficient Distributor RBP balance"
          }
        }
      * Skip to next bill

8.2 External API Call for Point Transfer:
    - Initialize tracking variables:
      * apiSuccess = false
      * apiResponse = null

    - Make API call to external retailer system:
      axios.post(RBP_POINT_CREDIT_API, {
        outlet_id: retailerUID,
        amount: totalRewardPoints,
        remarks: `Reward points for Bill no ${bill.billNo} for Retailer UID ${retailerUID} and DB Code ${dbCode}`,
        type: "SALES"
      })

    - Handle API Response:
      * If successful (!earnPointsResponse.data?.error):
        apiSuccess = true
      * If failed:
        apiResponse = earnPointsResponse.data

    - Handle API Errors:
      * Catch any network/API errors
      * apiResponse = err?.response?.data || err.message

8.3 Create RBP Transaction Record:
    - Regardless of API success/failure, create transaction:
      {
        distributorId: req.user._id,
        transactionType: "debit",
        transactionFor: "SALES",
        point: totalRewardPoints,
        balance: apiSuccess ? (latestBalance - totalRewardPoints) : latestBalance,
        billId: bill._id,
        retailerId: bill.retailerId,
        status: apiSuccess ? "Success" : "Failed",
        remark: `Reward points for Bill no ${bill.billNo}...`,
        apiResponse: apiSuccess ? null : apiResponse
      }

RBP SUMMARY:
- Points are only deducted if API call succeeds
- Failed transactions are logged with full error details  
- Distributor balance is protected from invalid deductions
- Complete audit trail for all point transfers

------------------------------------------------------------
STEP 9: FINAL RESPONSE
------------------------------------------------------------
9.1 Success Response:
    - After processing all bills, return:
      HTTP 200 OK
      {
        error: false,
        message: "Bills Delivered and inventory updated successfully",
        data: billStatus
      }

9.2 Error Handling:
    - All operations are wrapped in try-catch
    - Any unhandled error returns:
      HTTP 500 Internal Server Error
      {
        error: true,
        message: error.message || "Internal Server Error"
      }
    - Errors are also logged to console for debugging

CRITICAL BUSINESS IMPACT
------------------------------------------------------------
This API function is crucial because it:

1. INVENTORY MANAGEMENT:
   - Converts reserved stock to sold stock
   - Maintains accurate stock levels and valuations
   - Creates complete audit trail of stock movements

2. FINANCIAL ACCOUNTING: 
   - Records sales transactions in retailer ledger
   - Handles credit adjustments properly
   - Maintains accurate account balances

3. REWARD SYSTEM INTEGRATION:
   - Transfers reward points from distributor to retailer
   - Maintains distributor point balance accuracy
   - Integrates with external retailer reward API

4. DATA INTEGRITY:
   - Updates multiple related collections atomically
   - Prevents negative stock situations
   - Maintains referential integrity between bills, inventory, and transactions

4. Insufficient RBP Balance: Creates failed transaction, continues processing
5. External API Failure: Records failure, maintains distributor balance
6. Database Errors: Returns 500 with error logging
7. Race Conditions: Uses delays and sequential processing to prevent
