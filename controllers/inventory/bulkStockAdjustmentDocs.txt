---
FUNCTION NAME:

bulkStockAdjustment

DESCRIPTION:

This API endpoint processes stock quantity adjustments in bulk for distributors based on a CSV file upload. It reads adjustment data from a remote CSV, validates every row, updates the corresponding product inventory quantities (salable, unsalable, and offer stock), records stock transactions, and updates distributor reward points based on adjustments.

The function ensures system-wide synchronization between Inventory, Product, Transaction, Distributor, and DistributorTransaction models while providing auditability and error tolerance.

STEP 6: Calculate Adjustment Reward Points

Logic:

- Each product’s base_point is used to calculate changes in distributor’s reward points.

const basePoint = Number(product.base_point) || 0;
if (basePoint > 0) {
  let productAdjustmentPoints = basePoint * qty;
  if (adjustmentType === "add")
    totalAdjustmentPoints += productAdjustmentPoints;
  else if (adjustmentType === "reduce")
    totalAdjustmentPoints -= productAdjustmentPoints;
}

Explanation:

- Add Adjustment: Credits points (stock added).

- Reduce Adjustment: Debits points (stock reduced).

- Points accumulate in totalAdjustmentPoints.

The logic enables tracking of whether the distributor gains or loses reward points following stock adjustments.

STEP 9: Create DistributorTransaction for RBP Points


Condition:

If the distributor has RBP scheme mapping and total adjustment points ≠ 0.

const distributor = await Distributor.findById(distributorId).lean();
if (distributor?.RBPSchemeMapped === "yes" && totalAdjustmentPoints !== 0) {
  const latestTransaction = await DistributorTransaction.findOne({
    distributorId
  }).sort({ createdAt: -1 });

  const currentBalance = latestTransaction ? Number(latestTransaction.balance) : 0;
  const transactionType = totalAdjustmentPoints > 0 ? "credit" : "debit";
  const pointsToRecord = Math.abs(totalAdjustmentPoints);

  const newBalance = transactionType === "credit"
    ? currentBalance + pointsToRecord
    : Math.max(currentBalance - pointsToRecord, 0);
}

Create Distributor Transaction Record:

const distributorTransaction = new DistributorTransaction({
  distributorId,
  transactionType,
  transactionFor: "Adjustment Point",
  point: Math.round(pointsToRecord),
  balance: newBalance,
  status: "Success",
  remark: `Stock adjustment points for ${processedProducts.length} products for DB Code ${distributor.dbCode} via CSV adjustment`,
});
await distributorTransaction.save();

Purpose of Points Transaction:

- Reflects how distributor’s points balance changes due to stock additions or deductions.

- Ensures reward system consistency.

STEP 12: Comprehensive Error Handling


Each operational block is wrapped in multiple try-catch layers to ensure fault tolerance.

Possible Errors and Respective Handling:


1. Invalid CSV URL → 400 Bad Request

2. Download Failure → 500 Internal Server Error

3. CSV Parsing Errors → 500 Internal Server Error

4. Database Connectivity or Insert Failures → Logged, not fatal to global process.

5. Price or Product Missing Cases → Marked as skipped rows.

6. RBP Calculation Failure → Logged silently to prevent blocking the process.

Final catch:
catch (error) {
  console.error("Error in bulk stock adjustment:", error.message);
  res.status(500).json({ message: error.message });
}
Ensures every exception is caught and responded uniformly.

STEP 13: BUSINESS OBJECTIVES AND IMPACT


Primary Objectives:


1. Provide a seamless mechanism for distributors to bulk-adjust stock quantities.

2. Maintain accurate and traceable inventory adjustments.

3. Integrate adjustments automatically with distributor reward points through RBP (Reward Bonus Points).

Impact:

- Reduces manual adjustment time for large stock updates.

- Maintains distributor point integrity.

- Offers a transparent and auditable record of every adjustment.

STEP 14: FINAL OUTCOME


When executed successfully:


1. CSV file is downloaded, parsed, and processed efficiently.

2. Each valid product’s inventory is updated according to adjustment rules.

3. Corresponding transaction records are created for each change.

4. Distributor reward points (RBP) are adjusted to reflect net stock change.

5. Summary response shows details of successful adjustments and skipped entries.

6. Temporary files are deleted post-processing.

7. Entire flow supports partial success (valid rows processed even if others fail).

SUMMARY OF LOGICAL FLOW

1. Validate request input for csvUrl.

2. Download CSV and save it locally.

3. Parse CSV rows and validate individual entries.

4. Verify product, price, and inventory existence.

5. Adjust inventory stock quantities.

6. Calculate total point impact (Add = credit, Reduce = debit).

7. Record all stock transactions in bulk.

8. Adjust distributor reward points based on the total points.

9. Clean up CSV file.

10. Send summarized response with details.

11. Handle all errors gracefully.


IMPORTANT BUSINESS RULES

1. Only add or reduce adjustment types are valid inputs.

2. Only salable, unsalable, and offer stock types are allowed.

3. Each product in the CSV must exist in the system prior to adjustment.

4. Inventory entries must already exist for the product and distributor.

5. Points are adjusted only if the distributor is mapped under RBP scheme.

6. The entire CSV may contain valid and invalid rows; partial processing is supported.

7. Adjustments that reduce stock cannot reduce quantity below zero.

8. Price and base point values are mandatory for accurate point calculations.

9. File cleanup is performed after every operation, even on error.

10. Distributor balance will never become negative during point debit operations.
