PURPOSE: Step-by-step documentation of the createPurchaseReturn API function.

---
FUNCTION NAME:

createPurchaseReturn
---
DESCRIPTION:
This API endpoint is responsible for creating a Purchase Return entry for a given invoice. It validates the invoice and distributor, checks RBP (Reward Bonus Points) scheme mapping, ensures the distributor has sufficient points balance (when applicable), generates a unique Purchase Return code, and updates the invoice and distributor transactions accordingly.
The function maintains consistency between PurchaseReturn, Invoice, Distributor, and DistributorTransaction models.
---

STEP 1: Extract Request and Distributor Identification

- 
Extract the distributorId from the authenticated user:

const distributorId = req?.user?._id;

- 
Extract the invoiceId from req.body.

- 
Fetch the invoice document using: Invoice.findById(invoiceId)

- 
If invoice not found:

→ HTTP 404 Not Found

→ Error: "Invoice not found"

---

STEP 2: Validate Distributor Existence

- 
Fetch the distributor document:

Distributor.findById(distributorId).lean()

- 
If distributor not found:

→ HTTP 404 Not Found

→ Error: "Distributor not found"

---

STEP 3: Extract Total Base Points and Scheme Mapping

- 
Extract totalBasePoints from req.body (default to 0):

const totalBasePoints = req.body.totalBasePoints || 0;

- 
Check distributor’s RBP scheme mapping through distributor.RBPSchemeMapped.
---

STEP 4: Validate RBP Point Balance (Conditional)
- 
Condition: Only executed if totalBasePoints > 0 AND distributor.RBPSchemeMapped === "yes".

Steps:
	1. Retrieve the most recent DistributorTransaction record:

DistributorTransaction.findOne({ distributorId }).sort({ createdAt: -1 })

	2. Derive the current balance:

const currentBalance = Number(latestDbTransaction?.balance) || 0;

	3. Compare with required points:
		- If currentBalance < totalBasePoints →

→ HTTP 400 Bad Request

→ Error: "Insufficient point balance for purchase return. X points available, Y required."

- 
Else Case:

If totalBasePoints > 0 but RBP scheme is not mapped, the validation is skipped with a log message.

STEP 5: Generate Purchase Return Code and Create Record

- 


Generate a unique purchase return code using utility:

await generatePurchaseReturnCode("INV-RET")

- 
Create a new record in the PurchaseReturn collection:
const purchaseReturn = await PurchaseReturn.create({
  ...req.body,
  distributorId,
  code: await generatePurchaseReturnCode("INV-RET"),
});

- If creation fails → HTTP 400 Bad Request

→ Error: "Purchase return creation failed"

STEP 6: Update Associated Invoice

- Append the newly created purchase return ID to the invoice’s purchaseReturnIds array:
invoice.purchaseReturnIds = [
  ...(invoice.purchaseReturnIds || []),
  purchaseReturn._id,
];

- Save the updated invoice:

const updatedInvoice = await invoice.save();

This links the purchase return with its respective invoice for traceability.

STEP 7: Record Distributor Transaction (Conditional)
- 
Condition:

Execute only if totalBasePoints > 0 and distributor.RBPSchemeMapped === "yes".

- 
Create a debit transaction to deduct the used points:

const data = {
  distributorId,
  transactionType: "debit",
  transactionFor: "Purchase Return",
  point: Number(totalBasePoints),
  balance: Number(currentBalance) - Number(totalBasePoints),
  status: "Success",
  purchaseReturnId: purchaseReturn._id,
  remark: `Points deducting for purchase return with code ${purchaseReturn.code} for DB Code ${distributor.dbCode}`,
};

await DistributorTransaction.create(data);

- 

This ensures proper deduction and accounting of points used under the distributor’s RBP scheme.

- 
Else Case:

If RBP scheme not mapped, transaction is skipped with a log entry.

STEP 8: Send Final Response

- After successful completion of all steps, send a success response:

HTTP 201 Created

Response example:
{
  "error": false,
  "message": "Purchase return created successfully",
  "data": { ...purchaseReturnData },
  "invoice": { ...updatedInvoiceData }
}

STEP 9: Error Handling

- The entire logic is enclosed within a try-catch block.

- In case of an error, respond with:

→ HTTP 400 Bad Request or HTTP 404 (as appropriate)

→ throw error for Express middleware to handle.

FINAL OUTCOME

When executed successfully:

1. A new purchase return record is created in the database.

2. The purchase return is linked to its corresponding invoice.

3. RBP points validation ensures integrity of distributor balances.

4. Distributor’s transaction ledger is updated if RBP mapping exists.

5. A detailed success response is sent back to the client.


---

SUMMARY OF LOGICAL FLOW

1. Validate invoice existence.

2. Validate distributor existence.

3. Check totalBasePoints and RBP mapping.

4. Ensure sufficient RBP balance (if applicable).

5. Generate unique purchase return code.

6. Create purchase return record.

7. Link the record with the invoice.

8. Adjust distributor transactions if applicable.

9. Return success response.


---

IMPORTANT BUSINESS RULES

1. A valid invoice must exist before creating a purchase return.

2. Distributor must be valid and authenticated.

3. Points validation is mandatory only when:
	- totalBasePoints > 0

	- RBP Scheme is mapped (RBPSchemeMapped = "yes")

4. The system automatically deducts points through a debit transaction.

5. Each purchase return must have a unique, sequential code.

6. All errors are handled gracefully through consistent status codes and messages.

7. Purchase Return maintains traceability via its invoice linkage.
